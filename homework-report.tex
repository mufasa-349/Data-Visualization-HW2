\section{Gestalt and Bertin analysis (baseline --- before revision)}
\label{sec:analysis}

This section summarizes the written analysis in \texttt{task1-b-gestalt-bertin-analysis.txt}, describing the \textbf{pre-revision} interface.

\subsection{Gestalt --- strengths}
\begin{itemize}[leftmargin=*]
  \item \textbf{Proximity:} Filters are grouped in the sidebar; KPI metrics sit in one row, so controls and summary numbers form coherent units.
  \item \textbf{Similarity:} Both charts use the same hue family (blue) for data, signaling one phenomenon (immigration counts and their change).
  \item \textbf{Continuity:} The line chart supports a clear temporal reading from 1950 to 2005.
  \item \textbf{Figure--ground:} On a dark background, light chart elements and text separate well from the panel.
  \item \textbf{Common region:} The main content is visually separated from the sidebar (controls vs.\ readout).
\end{itemize}

\subsection{Gestalt --- weaknesses}
\begin{itemize}[leftmargin=*]
  \item \textbf{Similarity / semantics:} Red accents (sliders, active tab) vs.\ green KPI ``delta'' mix navigation and magnitude cues; users may briefly confuse ``red'' with negative data.
  \item \textbf{Continuity / YoY:} The YoY chart omits the first year in the filtered range (no prior row); a short caption was missing.
  \item \textbf{Proximity:} Four KPIs in one row can feel dense on narrow layouts; definitions were not always explicit.
  \item \textbf{Closure:} The total series could use a subtle area under the line to reinforce cumulative ``volume'' over time.
\end{itemize}

\subsection{Bertin --- strengths}
\begin{itemize}[leftmargin=*]
  \item \textbf{Position:} Time on the horizontal axis and magnitude on the vertical axis follows effective encoding priority.
  \item \textbf{Size (typography):} Title, KPI numerals, and axis ticks establish a reasonable hierarchy.
  \item \textbf{Color value:} Light blue on dark background works for the main series and bars.
\end{itemize}

\subsection{Bertin --- weaknesses}
\begin{itemize}[leftmargin=*]
  \item \textbf{Redundant encoding:} YoY sign was carried mainly by position above/below zero; hue could reinforce positive vs.\ negative years.
  \item \textbf{Mark size / noise:} Markers at every year on the line chart add visual noise on long series.
  \item \textbf{Units and definitions:} Axis titles and subtitles should state units and the meaning of \texttt{total} explicitly.
  \item \textbf{Contrast:} Axis tick labels on dark theme should meet comfortable contrast (WCAG-oriented).
\end{itemize}

\subsection{Summary (pre-revision)}
Strengths included clear temporal layout (Bertin: position), coherent grouping (Gestalt: proximity), and strong figure--ground separation. Gaps included redundant encoding for signed YoY change, clearer semantics for accent colors, and richer annotations (units, first-year YoY behavior).

\section{What is better after principles integration and revision?}
\label{sec:compare}

The revisions were not only cosmetic: they address the gaps identified in Section~\ref{sec:analysis} and make the dashboard \textbf{easier to read}, \textbf{less ambiguous}, and \textbf{more accessible} on a dark theme. Table~\ref{tab:checklist} lists \emph{what} changed; here we state \emph{why} the post-revision interface is stronger.

\subsection{User-visible improvements (benefits)}
\begin{itemize}[leftmargin=*]
  \item \textbf{Faster reading of the main trend (Gestalt closure):} The filled area under the annual total reinforces ``volume over time'' at a glance; removing dense per-year point markers reduces clutter so the long-run pattern stands out.
  \item \textbf{Clearer year-over-year story (Bertin redundant encoding):} Positive and negative changes are distinguishable both by bar direction \emph{and} by color, so users do not rely on a single channel; the zero baseline is visually sharper.
  \item \textbf{Less confusion between UI chrome and data (Gestalt similarity):} Aligning Streamlit's primary accent with the chart palette (\texttt{config.toml}) reduces the chance that control colors are mistaken for ``bad'' or ``negative'' data, compared with a stronger mismatch between red UI accents and green KPI deltas.
  \item \textbf{Explicit meaning of numbers (annotation):} Chart subtitles, axis titles with units (\emph{count per year}), KPI tooltips, and the YoY caption explain what \texttt{total} means and why the first filtered year has no YoY bar---reducing misinterpretation.
  \item \textbf{More comfortable reading on dark backgrounds (contrast):} Systematic tick and title colors in \texttt{apply\_dashboard\_theme()} improve legibility of axes and legends relative to an ad hoc default styling.
  \item \textbf{Coherent visual language across tabs:} One theme function and shared palette logic (continents, heatmap colorbar) make the dashboard feel like one instrument rather than unrelated charts.
\end{itemize}

\subsection{Mapping principles to outcomes}
\begin{itemize}[leftmargin=*]
  \item \textbf{Gestalt:} Revisions strengthened \textbf{similarity} (UI vs.\ data color harmony), \textbf{closure} (area fill for totals), and \textbf{proximity of meaning} (subtitles and help near the metrics they explain). Layout grouping was already sound and was preserved.
  \item \textbf{Bertin:} \textbf{Position} for time and magnitude was kept; \textbf{redundant encoding} was added for YoY sign; \textbf{color value} and tick contrast were standardized; unnecessary \textbf{mark} noise on the line chart was removed. Definitions and units are now tied explicitly to encodings (labels and tooltips).
\end{itemize}

\section{Conclusion (Task 1)}

We built an interactive Japan inbound immigration dashboard in Python (Streamlit + Plotly), analyzed it with Gestalt and Bertin lenses, and applied a focused set of revisions. After integration of these principles, the interface improves in \textbf{interpretive clarity} (what each number means), \textbf{encoding redundancy} for signed change (YoY), \textbf{visual continuity} for the level series (area fill), \textbf{legibility} (contrast and typography), and \textbf{consistency} across charts and the Streamlit shell. The screenshots \texttt{task1-a.png} and \texttt{task1-b.png} document the interface before and after these changes; Section~\ref{sec:compare} summarizes concrete benefits of the revised version.

\part{Task 2 --- Whoop-style fitness dataset}

\section{Task 2 Part A: Analysis of the Whoop health dataset (for a mobile app)}
\label{sec:whoop}

This section looks at \path{whoop_fitness_dataset_100k.csv}. The file has one row per user per day (about 100k rows). It is built like a \textbf{wearable / Whoop-style} daily log: sleep, recovery, strain, workouts, and heart metrics. Below I explain what is in the data and what I would actually show in a \textbf{phone app} and why.

\subsection{What the dataset contains (big picture)}
Each row is a \textbf{single day} for one \texttt{user\_id}. You get the date, day of week, some \textbf{profile} fields (age, gender, height, weight), and then numbers that describe \textbf{readiness}, \textbf{load}, \textbf{sleep}, and \textbf{activity}. Workout-related columns are filled when the person trained; on rest days the activity type is often ``Rest Day'' and duration/strain can be zero.

\subsection{Column groups (simple overview)}
\begin{itemize}[leftmargin=*]
  \item \textbf{Time and identity:} \texttt{user\_id}, \texttt{date}, \texttt{day\_of\_week} --- needed to sort days and build a calendar or timeline in the app.
  \item \textbf{Profile:} \texttt{age}, \texttt{gender}, \texttt{weight\_kg}, \texttt{height\_cm}, \texttt{fitness\_level}, \texttt{primary\_sport} --- good for onboarding screens, goals, and simple personalization (not something that changes every hour).
  \item \textbf{Daily readiness / load (``Whoop-like'' scores):} \texttt{recovery\_score}, \texttt{day\_strain} --- these are the numbers users often want \textbf{on the home screen}: ``how ready am I?'' vs.\ ``how hard was today?''.
  \item \textbf{Sleep:} \texttt{sleep\_hours}, \texttt{sleep\_efficiency}, \texttt{sleep\_performance}, \texttt{light\_sleep\_hours}, \texttt{rem\_sleep\_hours}, \texttt{deep\_sleep\_hours}, \texttt{wake\_ups}, \texttt{time\_to\_fall\_asleep\_min} --- perfect for a \textbf{sleep tab}: total time, quality, and stage breakdown.
  \item \textbf{Heart and stress signals:} \texttt{hrv}, \texttt{resting\_heart\_rate}, \texttt{hrv\_baseline}, \texttt{rhr\_baseline}, \texttt{respiratory\_rate}, \texttt{skin\_temp\_deviation} --- useful for ``recovery'' and health trends; comparing today to \textbf{baseline} helps users see if they are off their normal (tired, sick, etc.).
  \item \textbf{Energy:} \texttt{calories\_burned} --- easy to show as a daily ring or total, similar to other fitness apps.
  \item \textbf{Workout block:} \texttt{workout\_completed}, \texttt{activity\_type}, \texttt{activity\_duration\_min}, \texttt{activity\_strain}, \texttt{avg\_heart\_rate}, \texttt{max\_heart\_rate}, \texttt{activity\_calories}, \texttt{hr\_zone\_1\_min}--\texttt{hr\_zone\_5\_min}, \texttt{workout\_time\_of\_day} --- detail screen after a session: what sport, how long, how hard, and time in heart-rate zones.
\end{itemize}

\subsection{Which attributes are most useful in a mobile app (and why)}
\begin{itemize}[leftmargin=*]
  \item \textbf{\texttt{recovery\_score} + \texttt{day\_strain}:} Very useful. On a small screen you want \textbf{two clear numbers} that answer: ``Can I train hard today?'' and ``Did I already load my body?'' They match how real wearables market their product, so users understand them quickly.
  \item \textbf{Sleep totals and stages (\texttt{sleep\_hours}, efficiency, light/REM/deep):} Very useful. People open health apps in the morning to see \textbf{how long} they slept and \textbf{sleep quality}. Stages help users see if deep/REM was short even when total hours look okay.
  \item \textbf{\texttt{hrv} and \texttt{resting\_heart\_rate} with baselines:} Useful for a ``recovery'' or ``body status'' card. If today's HRV is far from \texttt{hrv\_baseline} or RHR is off \texttt{rhr\_baseline}, the app can show a gentle warning (rest day, not a medical diagnosis---just trend).
  \item \textbf{\texttt{calories\_burned}:} Useful for users who care about energy balance; easy to show as one number or a progress ring.
  \item \textbf{Workout summary (\texttt{activity\_type}, \texttt{activity\_duration\_min}, \texttt{activity\_strain}, HR avg/max, \texttt{activity\_calories}):} Essential on the \textbf{workout detail} page. The user wants to see what they did and how hard it was without scrolling forever.
  \item \textbf{Heart-rate zones (\texttt{hr\_zone\_1\_min}--\texttt{hr\_zone\_5\_min}):} Useful for people who train with intensity, but on mobile it is better to show a \textbf{small bar chart or one line of text} (e.g.\ ``12 min hard'') instead of five raw numbers, so the screen does not feel crowded.
  \item \textbf{\texttt{workout\_time\_of\_day}:} Nice for habits---morning vs.\ evening training---and for simple insights (``you usually work out in the evening'').
  \item \textbf{Profile fields (\texttt{fitness\_level}, \texttt{primary\_sport}):} Good for recommendations and filtering (beginner vs.\ advanced), not for daily graphs.
\end{itemize}

\subsection{Attributes that are less critical on the phone (or need simplification)}
\begin{itemize}[leftmargin=*]
  \item \textbf{Five separate HR zone columns:} Important for training nerds, but the app should \textbf{aggregate} them for the main view (e.g.\ total moderate + hard minutes).
  \item \textbf{\texttt{skin\_temp\_deviation} and \texttt{respiratory\_rate}:} Still valuable for wellness, but many users will not look at them every day---better in a ``health details'' or weekly summary.
  \item \textbf{\texttt{wake\_ups} and \texttt{time\_to\_fall\_asleep\_min}:} Good for sleep geeks; keep them on the sleep detail page, not on the home screen.
\end{itemize}

\subsection{Things to be careful about (student-level notes)}
\begin{itemize}[leftmargin=*]
  \item The data is \textbf{not real medical data} from our class project point of view---treat it as a \textbf{sample} for building UI and charts.
  \item On \textbf{rest days}, workout fields are often zero or ``N/A''; the app must handle that so it does not look like a bug.
  \item \textbf{Privacy:} a real app would need strong rules for \texttt{user\_id} and health fields; we only talk about \textbf{which columns are useful for design}, not about storing real patient data.
\end{itemize}

\subsection{Short summary}
For a mobile app, I would prioritize \textbf{recovery}, \textbf{strain}, \textbf{sleep summary}, \textbf{HRV/RHR vs.\ baseline}, and \textbf{workout type + duration + strain + calories}. Heart-rate zones and extra sleep metrics are great on a second screen. Profile fields support onboarding and personalization. This matches what users expect from a Whoop-style experience: \textbf{simple daily scores first}, details when they tap.

\subsection{Mobile dashboard layout (four cards)}
The dashboard is split into four main \textbf{cards} the user would see when opening the app.

\begin{itemize}[leftmargin=*]
  \item \textbf{Daily overview (``Ready to train?'')}: The most important metrics---\textbf{Recovery Score} and \textbf{Day Strain}---sit at the top. They map to \texttt{recovery\_score} and \texttt{day\_strain} in the CSV.
  \item \textbf{Sleep analysis}: Shows sleep duration and quality with a \textbf{stacked} view of stages (light, REM, deep). This uses \texttt{sleep\_hours}, \texttt{sleep\_efficiency}, and stage columns such as \texttt{light\_sleep\_hours}, \texttt{rem\_sleep\_hours}, \texttt{deep\_sleep\_hours}.
  \item \textbf{Recovery trends}: A \textbf{line chart} of HRV over several days compared to the user's \textbf{baseline}. This ties to \texttt{hrv}, \texttt{hrv\_baseline}, and related day-by-day rows.
  \item \textbf{Activity summary}: Daily calories and \textbf{heart-rate zones} (time in each zone). This uses \texttt{calories\_burned} and \texttt{hr\_zone\_1\_min}--\texttt{hr\_zone\_5\_min}.
\end{itemize}

\subsection{Gestalt principles (how the sketch uses them)}
\begin{itemize}[leftmargin=*]
  \item \textbf{Proximity:} Numbers are grouped into clear cards. Sleep hours, efficiency, and stages stay in one block so the user reads them as one ``sleep'' unit.
  \item \textbf{Similarity:} One color family (blue shades) for sleep stages (Deep, REM, Light) so they are seen as parts of the same sleep story.
  \item \textbf{Continuity:} The HRV chart is a connected line so the eye follows the trend over time (up or down vs.\ scattered dots).
  \item \textbf{Figure--ground:} Large, high-contrast numbers for Recovery and Strain (e.g.\ ``89\%'') stand out as the main ``figure'' on a calm background---quick read in under a second.
\end{itemize}

\subsection{Bertin's visual variables (Semiology of Graphics)}
\begin{itemize}[leftmargin=*]
  \item \textbf{Position:} Vertical \textbf{order} = importance (best scores at the top). Horizontal \textbf{time} on the HRV strip = last 7 days.
  \item \textbf{Size:} Recovery and Strain digits are larger than other text, so the user knows what matters most for daily decisions.
  \item \textbf{Value / color:}
    \begin{itemize}
      \item \textbf{Qualitative (status):} Green for high recovery, orange/red for low---fast status read.
      \item \textbf{Sequential (intensity):} HR zone chart uses a yellow--to--red gradient: darker/warmer = harder effort.
    \end{itemize}
  \item \textbf{Shape / texture:} Solid line = current HRV; dashed line = baseline. Different texture so ``today'' vs.\ ``normal'' is clear without a heavy legend.
\end{itemize}

\subsection{Data mapping summary (what encodes what)}
\begin{itemize}[leftmargin=*]
  \item \textbf{Recovery \& Strain:} size and color (color = quick status).
  \item \textbf{Sleep stages:} bar length + blue hue steps (composition of the night).
  \item \textbf{HRV \& RHR:} \textbf{position} on a time axis (trend vs.\ baseline).
  \item \textbf{HR zones:} bar length + saturation (redder = harder zone).
\end{itemize}

Sketch~2 is a \textbf{dark-mode} layout aimed at high contrast in low light. It pulls in \textbf{more columns} from the CSV than sketch~1: breathing and skin temperature sit next to heart metrics, and baselines appear next to live values.

\paragraph{New features (data mapping).}
\begin{itemize}[leftmargin=*]
  \item \textbf{Health monitor card:} Combines \texttt{respiratory\_rate} and \texttt{skin\_temp\_deviation} with HR-style signals so the user sees a wider ``internal state'' in one place.
  \item \textbf{Baseline comparisons:} Today's HRV/RHR are shown with \texttt{hrv\_baseline} and \texttt{rhr\_baseline} as reference (small text or subscript). Bertin: \textbf{position} and alignment make ``now vs.\ normal'' easy to compare without a second chart.
  \item \textbf{Intensity (HR zones):} Uses \texttt{hr\_zone\_1\_min} through \texttt{hr\_zone\_5\_min} in a horizontal bar---left-to-right matches ``easy $\rightarrow$ hard'' reading.

\textbf{Closure:} Each topic lives in its own card (health vs.\ activity) so groups are clear without thick lines. \textbf{Figure--ground:} Neon-style greens/blues/reds on near-black make numbers jump forward. \textbf{Similarity:} Repeated pattern Value $\rightarrow$ Label $\rightarrow$ Baseline in the health strip teaches one rhythm. \textbf{Proximity:} Duration, avg HR, and max HR stay together as one workout event.

\paragraph{Bertin (short).}
Recovery/Strain stay \textbf{larger} than the monitor tiles (global vs.\ supportive metrics). \textbf{Hue} on zones runs green$\rightarrow$yellow$\rightarrow$orange$\rightarrow$red for aerobic$\rightarrow$anaerobic. \textbf{Orientation:} Horizontal zones save vertical space for labels. \textbf{Position:} Readiness block stays at the \textbf{top} (highest priority).

\subsubsection*{Key design enhancements}
\begin{itemize}[leftmargin=*]
  \item \textbf{Bertin --- size and orientation:} The \textbf{orbit} uses concentric circles: different \textbf{radius} = different metrics (Strain, Sleep, HRV, RHR), so the user can read ``one system'' at a glance. The \textbf{zone bar} uses a \textbf{linear} (horizontal) layout: left-to-right shows how much time fell in aerobic vs.\ anaerobic HR zones, so \textbf{proportion} of the workout is easy to see.
  \item \textbf{Gestalt --- proximity:} Health numbers sit in a \textbf{2$\times$2 grid}. HRV and its baseline are \textbf{close together} on the same card, so the brain treats them as one idea: ``current readiness vs.\ normal.''
  \item \textbf{Color context (similarity):} Card \textbf{left-border} colors are not random: they \textbf{match} the color of the matching ring in the orbit. Same color for the same metric = same mental group (Gestalt \textbf{similarity}).
  \item \textbf{Health trends (common region):} On the HRV card, a small green ``+1'' (or similar delta) sits \textbf{next to} the main number. That groups the \textbf{change} with the \textbf{value} in one visual region, so the user reads ``status + update'' as one chunk (Gestalt \textbf{common region}).
\end{itemize}
